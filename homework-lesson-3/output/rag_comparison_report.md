# Порівняльний аналіз підходів до побудови RAG: Naive, Sentence-Window та Parent-Child

Цей звіт містить порівняння трьох ключових стратегій сегментації (chunking) та пошуку в системах Retrieval-Augmented Generation (RAG). Вибір правильного підходу є критичним для балансу між точністю відповіді, контекстуальною повнотою та вартістю обчислень.

## 1. Огляд підходів

### Naive RAG (Базовий підхід)
Це найпростіша реаліза/застосування RAG, де документ розбивається на чанки фіксованого розміру (наприклад, за кількістю токенів або символів) з невеликим перекриттям (overlap).
- **Механіка:** Текст $\rightarrow$ Fixed-size chunks $\rightarrow$ Embedding $\rightarrow$ Vector Store $\rightarrow$ Retrieval.
- **Особливість:** Не враховує семантичну цілісність речень або структуру документа.

### Sentence-Window Retrieval (Пошук за вікном речень)
Більш просунутий метод, що фокусується на збереженні контексту навколо знайденого фрагмента.
- **Механіка:** Кожне речення зберігається як окремий векторний чанк для точного пошуку. Однак, під час генерації відповіді (generation phase), замість одного речення, LLM отримується "вікно" — поточне речення плюс кілька попередніх та наступних речень навколо нього.
- **Особливість:** Дозволяє знаходити дуже точні факти, але надає достатньо контексту для розуміння змісту.

### Parent-Child Retrieval (Ієрархічний підхід)
Стратегія, що використовує багаторівневу структуру даних для розділення процесу пошуку та процесу генерації.
- **Механіка:** Документ розбивається на великі "батьківські" чанки (Parent chunks) та дрібні "дитячі" чанки (Child chunks). Пошук відбувається за дрібними дитячиму чанками (що забезпечує високу точність/recall), але в контекст LLM передається повний батьківський чанк.
- **Особливість:** Використовує дрібні фрагменти для точного знаходження, але надає великий контекст для якісного синтезу тексту.

## 2. Порівняльна таблиця

| Характеристика | Naive RAG | Sentence-Window | Parent-Child |
| :--- | :--- | :--- | :--- |
| **Точність пошуку (Recall)** | Низька/Середня (може розривати думку) | Висока (фокус на окремих реченнях) | Висока (дрібні дитячі чанки) |
| **Повнота контексту** | Залежить від розміру чанку | Середня (обмежена "вікном") | Висока (повний батьківський чанк) |
| **Складність реалізації** | Дуже низька | Середня | Висока |
| **Витрати на токени (LLM)** | Прогнозовані | Можуть бути високими при великому вікні | Високі (передача великих батьківських чанків) |
мульти-рівневий пошук |

## 3. Практичні Trade-offs (Компроміси)

### Коли обирати Naive RAG:
*   **Прототипування:** Якщо потрібно швидко перевірити ідею.
*   **Однорідні дані:** Коли тексти мають дуже просту структуру та не містять складних логічних зв'язків.
*   **Обмежений бюджет:** Найменші витрати на індексацію та зберігання.

### Коли обирати Sentence-Window:
*   **Пошук фактів:** Коли важливо знайти конкретне твердження (наприклад, у юридичних документах або технічних специфікаціях).
*   **Баланс точність/контекст:** Якщо ви хочете уникнути "шуму" від великих чанків, але не можете дозволити собі втратити зв'язок між реченнями.

### Коли обирати Parent-Child:
*   **Складні документи:** Для підручників, довгих звітів або документації, де важлива ієрархія (розділ $\rightarrow$ підрозділ $\rightarrow$ абзац).
*   **Висока точність синтезу:** Коли критично важливо, щоб LLM бачила повну картину навколо знайденого факту, навіть якщо пошук був дуже специфічним.
*   **Ризик "галлюцинацій":** Цей метод найкраще бореться з галюцинаціями, спричиненими неповним контекстом.

## 4. Висновки та обмеження
Підхід **Naive RAG** часто стає причиною поганої якості відповідей через розрив семантичних одиниць. Перехід до **Sentence-Window** або **Parent-Child** значно покращує Recall (здатність знаходити релевантне) та контекстуальну точність, проте це потребує складнішої інфраструктури для зберігання зв'язків між чанками та збільшує витрати на обробку токенів під час генерації.

**Обмеження дослідження:** Даний звіт базується на аналізі сучасних стратегій чанкування (2024-2025 рр.). Ефективність кожного методу сильно залежить від якості embedding-моделі та специфіки домену даних.
