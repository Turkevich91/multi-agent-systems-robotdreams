# Практики Prompt Engineering для Research Agent (ReAct)

## Висновок
Для ефективної роботи Research Agent, що використовує ReAct loop, необхідно застосовувати специфічні техніки Prompt Engineering, які забезпечують стабільність циклу "мислення-дії", точність використання інструментів та надійність фінального синтезу.

## Ключові практики Prompt Engineering для агента

### 1. Few-Shot Prompting (Навчання на прикладах)
Це найважливіша практика для ReAct. Агент повинен бачити кілька прикладів того, як саме він має формулювати `Thought`, `Action` та як інтерпретувати `Observation`.
* **Навіщо:** Без прикладів модель може почати писати довгі роздуми замість чітких команд для інструментів або ігнорувати формат виклику функцій.

### 2. Chain-of-Thought (CoT) Integration
Хоча ReAct — це окремий патерн, він базується на CoT. Агент повинен бути змушений спочатку генерувати `Thought` перед кожною дією.
* **Навіщо:** Це дозволяє моделі "проговорити" логіку планування, що зменшує ймовірність випадкового виклику неправильного інструмента (наприклад, пошук замість читання URL).

### 3. Role Prompting (Визначення ролі)
Чітке визначення ідентичності ("Ти — дослідницький агент...") та меж (Safety boundaries).
* **Навіщо:** Це встановлює контекст відповідальності та запобігає "виходу за межі" (наприклад, спробам змінити системні інструкції через вміст вебсторінок).

### 4. Output Structuring & Constraint Enforcement
Використання жорстких форматів для відповідей (Markdown, JSON або специфічні теги `Action:`, `Observation:`).
* **Навіщо:** Це дозволяє програмному коду (Parser) розрізняти текст, який призначений для користувача, і команди, які мають бути передані в інструменти.

### 5. Self-Reflection & Verification (Self-Correction)
Інструктування агента проводити перевірку результатів перед завершенням (`Final Answer`).
* **Навіщо:** Це змушує модель порівнювати отримане `Observation` із початковим запитом і перевіряти, чи достатньо зібрано даних для відповіді на всі частини питання.

## Порівняльна таблиця технік у контексті агента

| Техніка | Застосування в ReAct Agent | Результат |
| :---SS| :--- | :--- |
| **Few-Shot** | Надання прикладів циклів `Thought -> Action`. | Стабільний синтаксис виклику інструментів. |
| **CoT** | Обов'язковий крок `Thought` перед дією. | Логічна послідовність та планування. |
| **Role Prompting** | Встановлення правил дослідження та етики. | Дотримання меж безпеки та формату звіту. |
| **Self-Reflection** | Крок перевірки даних перед `write_report`. | Висока якість та повнота фінального звіту. |

## Обмеження
Ефективність цих практик критично залежить від розміру контекстного вікна. Занадто велика кількість прикладів (Few-Shot) або занадто довгі роздуми (CoT) можуть призвести до того, що агент "забуде" початкову інструкцію або вичерпає ліміт токенів до завершення дослідження.
