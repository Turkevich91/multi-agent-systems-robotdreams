# Дослідження: ReAct Loop vs Chain-of-Thought (CoT)

## Короткий висновок
Цей звіт пояснює механізм роботи **ReAct loop** (Reasoning + Acting) для дослідницьких агентів та проводить порівняння з методом **Chain-of-Thought (CoT)**. Основна відмінність полягає в тому, що ReAct дозволяє моделі взаємодіяти із зовнішнім світом через інструменти (tools), тоді як CoT обмежений лише внутрішніми знаннями моделі.

## Аналіз механізмів

### 1. Що таке ReAct loop?
ReAct — це фреймворк, який поєднує логічне міркування (**Reasoning**) та виконання дій (**Acting**). Процес працює як циклічний петля (loop), що складається з трьох ключових компонентів:

*   **Thought (Міркування):** Модель формулює внутрішній план, розбиває складну задачу на підзадачі, відстежує прогрес або планує наступний крок.
*   **Action (Дія):** Виклик зовнішнього інструменту (наприклад, `web_search`, `calculator` або `read_url`) для отримання нових даних.
*   **Observation (Спостереження):** Результат виконання дії (текст зі сторінки, результат обчислення), який повертається моделі та стає частиною контексту для наступного кроку "Thought".

Цей цикл повторюється до моменту, поки модель не отримає достатньо інформації для формування фінальної відповіді.

### 2. Порівняння: ReAct vs Chain-of-Thought (CoT)

| Характеристика | Chain-of-Thought (CoT) | ReAct Loop |
| :--- | :--- | :--- |
| **Принцип роботи** | Послідовна генерація логічних кроків на основі лише внутрішніх ваг моделі. | Цикл "міркування $\rightarrow$ дія $\rightarrow$ спостереження" з використанням зовнішніх інструментів. |
| **Доступ до знань** | Обмежений датою завершення навчання моделі (static knowledge). | Динамічний: може отримувати актуальну інформацію через пошук (dynamic knowledge). |
| **Ризик галюцинацій** | Високий при роботі з фактами, яких немає в тренувальних даних. | Нижчий: модель може перевірити свої припущення через фактичні спостереження. |
| **Взаємодія** | Пасивний (тільки генерація тексту). | Активний (взаємодія з API, базами даних, веб-сторінками). |
| **Складність** | Простіший у реалізації (просто промпт "think step by step"). | Складніший (потребує налаштування інструментів та обробки результатів). |

## Практичні trade-offs (Компроміси)

*   **Використання ReAct:**
    *   *Плюси:* Висока точність у задачах, що потребують перевірки фактів; здатність вирішувати багатоетапні задачі (multi-hop reasoning).
    *   *Мінуси:* Вища затримка (latency) через очікування відповіді від інструментів; вища вартість (token cost) через розширення контексту спостереженнями.
*   **Використання CoT:**
    *   *Плюси:* Швидкість; мінімальна вартість; ідеально для задач на чисту логіку або математику, де не потрібен зовнішній пошук.
    *   *Мінуси:* Нездатність оновити знання; схильність до "впевнених" помилок (hallucinations).

## Обмеження дослідження
Даний звіт базується на аналізі структури ReAct та порівняльних характеристик, викладених у доступних технічних джерелах. Дослідження не включало емпіричного тестування швидкості (latency) на конкретних моделях, а зосереджувалося на архітектурній різниці між підходами.
