# Дослідження: ReAct Loop vs Chain-of-Thought

## Короткий висновок
Цей звіт пояснює механізм роботи **ReAct (Reasoning and Acting)** для дослідницьких агентів та проводить порівняння з методом **Chain-of-Thought (CoT)**. Основна відмінність полягає в тому, що ReAct інтегрує зовнішнє взаємодія з оточенням (інструментами) у процес міркування, тоді як CoT обмежений лише внутрішніми знаннями моделі.

## Аналіз механізмів

### 1. Що таке ReAct Loop?
ReAct — це фреймворк, який поєднує текстове міркування з виконанням конкретних дій. Процес працює за циклом:
*   **Thought (Міркування):** Модель формулює план або аналізує поточний стан задачі (декомпозиція цілі, відстеження прогресу).
*   **Action (Дія):** Виклик зовнішнього інструменту (пошук у Google, запит до бази даних, калькулятор).
*   **Observation (Спостереження):** Отримання результату від інструменту (текст зі сторінки, число, помилка), який стає новим контекстом для наступного кроку "Thought".

Цей цикл дозволяє агенту коригувати свій шлях на основі отриманих фактичних даних.

### 2. Порівняння: ReAct vs Chain-of-Thought (CoT)

| Характеристика | Chain-of-Thought (CoT) | ReAct Loop |
| :--- | :--- | :--- |
| **Природа процесу** | Статичний (внутрішній) | Динамічний (зовнішній + внутрішній) |
| **Джерело знань** | Тільки параметрична пам'ять моделі | Параметрична пам'ять + зовнішні інструменти |
| **Здатність до перевірки** | Не може перевірити факти, що призводить до галюцинацій | Може перевіряти твердження через пошук (Grounding) |
| **Робота з новими даними** | Обмежена датою завершення навчання моделі | Доступ до актуальної інформації в реальному часі |
| **Складність задач** | Добре підходить для логічних та математичних задач | Ідеально для дослідницьких завдань (Research) |

### 3. Практичні Trade-offs (Компроміси)

*   **Переваги ReAct:** Зменшення галюцинацій, можливість вирішення багатоетапних задач, що потребують пошуку, висока адаптивність до помилок інструментів.
*   **Недоліки ReAct:** Вища затримка (latency) через необхідність очікування відповідей від інструментів, вища вартість (більша кількість токенів на кожен крок циклу), складність у розробці стабільного парсингу дій.

## Обмеження дослідження
Дане дослідження базується на аналізі доступних технічних статей та документації щодо Prompt Engineering. Результати можуть змінюватися залежно від архітектури конкретної LLM (наприклад, використання Function Calling у сучасних моделях робить ReAct більш стабільним).
