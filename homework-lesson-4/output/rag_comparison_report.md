# Порівняння стратегій RAG: Naive RAG, Sentence-Window Retrieval та Parent-Child Retrieval

## Короткий висновок
Цей звіт представляє порівняльний аналіз трьох основних підходів до ретривалу (retrieval) у системах RAG. Основна мета переходу від Naive RAG до складніших методів поляга/гає полягає у вирішенні дилеми "точність проти контексту": як знайти максимально точну відповідь, не втративши при цьому необхідного фонового контексту для LLM.

## Аналіз стратегій

### 1. Naive RAG (Базовий підхід)
Найпростіша реалізація, де документ розбивається на чанки (chunks) фіксованого розміру, які зберігаються у векторній базі даних.
- **Принцип роботи:** Пошук за семантичною схожістю $\rightarrow$ отримання найбільш схожих чанків $\rightarrow$ передача їх в LLM.
- **Переваги:** Максимальна простота реалізації та низька вартість обробки.
- **Недоліки:** 
    - **Проблема фрагментації:** Якщо важливий факт розірвано межею чанку, контекст втрачається.
    - **Шум:** Великі чанки можуть містити багато нерелевантної інформації (false positives), що заплутує модель.

### 2. Sentence-Window Retrieval (Метод віконного ретривалу)
Стратегія, що фокусується на використанні маленьких одиниць для пошуку та великих — для генерації.
- **Принцип роботи:** Документ індексується на рівні окремих речень (або дуже малих чанків). Під час пошуку знаходяться найбільш релевантні речення, але замість них у LLM передається "вікно" — обране речення разом із кількома сусідніми реченнями (preceding/succeeding context).
- **Переваги:** 
    - Висока точність пошуку завдяки мінімізації шуму в індексі.
    - Збереження контексту за рахунок розширення вікна під час генерації.
- **Недоліки:** Потребує складнішої структури метаданих (відстеження порядкового номера речень у документі).

### 3. Parent-Child Retrieval (Auto-merging / Hierarchical Retrieval)
Більш ієрархічний підхід, що базується на зв'язках між різними рівнями чанківгу.
- **Принцип роботи:** Документ ділиться на великі "батьківські" чанки (parent chunks), які потім розбиваються на менші "дитячі" чанки (child chunks). Пошук виконується по дитячих чанках, але коли знайдено релевантний "дитина", система автоматично підтягує повний "батьківський" контекст.
- **Переваги:** 
    - Дозволяє ефективно використовувати малі чанки для точного семантичного пошуку.
    - Забезпечує LLM повною інформацією (батьківським контекстом) для розуміння глибини теми.
- **Недоліки:** Висока складність індексації та необхідність підтримки ієрархічних зв'язків у базі даних.

## Порівняльна таблиця trade-offs

| Характеристика | Naive RAG | Sentence-Window | Parent-Child |
| :--- | :--- | :--- | :--- |
| **Точність пошуку** | Низька/Середня | Висока | Висока |
| **Повнота контексту** | Залежить від розміру чанку | Висока (динамічна) | Дуже висока |
| **Складність системи** | Мінімальна | Середня | Висока |
| **Ризик шуму** | Високий | Низький | Низький |

## Обмеження дослідження
- Дослідження базується на аналізі доступних технічних статей та документації. 
- Не було проведено емпіричних тестів (benchmarking) на конкретних датасетах; висновки базуються на теоретичних властивостях алгоритмів.
- Ефективність кожного методу сильно залежить від якості чанкування (chunking strategy) та якості embedding-моделі.
