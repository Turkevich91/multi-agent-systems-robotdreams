# Порівняння архітектур RAG: Naive, Sentence-Window та Parent-Child

**Коротке порівняння:**

| Метод | Точність контексту | Складність реалізації | Обчислювальні витрати |
| :--- | :--- | :--- | :--- |
| **Naive RAG** | Низька/Середня | Дуже низька | Низькі |
| **Sentence-Window** | Висока | Середня | Середні |
| **Parent-Child** | Найвища | Висока | Високі |

---

*Примітка щодо методології: Цей звіт був підготовлений за допомогою протоколу Plan -> Research -> Critique -> Report. Інструменти для пошуку та збереження даних були доступні через MCP (Model Context Protocol), а спеціалізовані агенти (Planner, Researcher, Critic) викликалися через делегування ACP (Agent Communication Protocol).*

## 1. Визначення методів

* **Naive RAG (Наївний RAG):** Найпростіша форма архітектури, де документ розбивається на однакові за розміром фрагменти (chunks). Кожен фрагмент індексується та використовується як єдине ціле: під час пошуку знаходять найбільш релевантний чанк, і він подається в LLM для генерації відповіді.
* **Sentence-Window Retrieval (Пошук за сенсовим вікном):** Метод, що фокусується на окремих реченнях. Під час індексації кожне речення розглядається як окремий об'єкт, але під час генерації відповіді система "розширює" знайденого речення, додаючи до нього сусідні речення (вікно контексту). Це дозволяє зберігати точність пошуку, не втрачаючи контексту.
* **Parent-Child Retrieval (Пошук «Батько-Дитина» / Auto-Merging):** Ієрархічний підхід, де дані організовані у два рівні. Маленькі фрагменти («діти») використовуються для точного семантичного пошуку, а великі фрагменти («батьки»), що містять ці «діти», використовуються для надання широкого контексту LLM. Якщо знайдено достатньо релевантних «дітей», система об'єднує їх у батьківський чанк.

## 2. Детальний аналіз архітектур

| Критерій | Naive RAG | Sentence-Window Retrieval | Parent-Child Retrieval |
| :--- | :--- | :--- | :--- |
| **Контекстуальна багатькість / Точність** | **Низька/Середня.** Ризик втрати контексту через розрив тексту на чанки або надмірний шум, якщо чанки занадто великі. | **Висока.** Забезпечує високу точну відповідність запиту завдяки пошуку по реченнях та багатий контекст завдяки вікну. | **Найвища.** Найкраще зберігає логічну цілісність документа завдяки ієрархічному об'єднанню фрагментів. |
| **Обчислювальні витрати (Indexing & Retrieval)** | **Низькі.** Мінімальна кількість операцій при індексації та швидкий пошук. | **Середні.** Потребує додаткових кроків для управління вікнами контексту під час ретривалу. | **Високі.** Складніша індексація (зберігання зв'язків між рівнями) та вища потреба в пам'яті/обробці при об'єднанні. |
| **Складність реалізації** | **Дуже низька.** Легко налаштувати за допомогою стандартних бібліотек (LangChain, LlamaIndex). | **Середня.** Потребує логіки для виділення речень та керування розміром вікна навколо них. | **Висока.** Вимагає створення складних структур даних (дерево або зв'язані списки) та логіки авто-злиття (auto-merging). |

## 3. Практичні компроміси (Trade-offs)

* **Naive RAG:** Обирайте для простих завдань, де швидкість та низька вартість є пріоритетом, а дані не мають складної структури.
* **Sentence-Window:** Обирайте, коли точність пошуку критична (наприклад, пошук конкретних фактів), але важливо, щоб модель розуміла оточення цього факту.
* **Parent-Child:** Обирайте для складних документів (юридичні контракти, наукові статті), де розуміння структури та взаємозв'язку між частинами тексту є ключовим для правильної відповіді.

## 4. Обмеження кожного підходу

* **Naive RAG:**
    * Втрата контексту при розриві важливих речень на межі чанків.
    * Проблема "шуму", якщо чанк занадто великий і містить нерелевантну інформацію.
* **Sentence-Window Retrieval:**
    * Можлива надмірна витрата токенів, якщо вікно контексту встановлено занадто великим.
    * Потребує якісного попереднього розділення тексту на речення (sentence splitting).
* **Parent-Child Retrieval:**
    * Значно складніша архітектура бази даних (потреба в підтримці зв'язків між чанками).
    * Вища затримка (latency) через необхідність обробки логіки злиття фрагментів.
